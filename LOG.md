- Explained `fitfroquad` (`general_model/Smooth/Quad.py`) to the user; no code changes. Key points: it fits q(y) = c + G.y + 0.5 y'Hy to the sample points and returns (G, H) at the origin of the given coordinates, dropping c. With m <= n+1 points it returns an affine fit with H = 0. Otherwise it projects out the affine-explainable part of f (Z = I - QQ' from QR of [1|Y]), least-squares the quadratic basis (phi2eval: half-squares and cross-terms/sqrt(2)) against the projected residual, then fits c and G to what remains. The 1/2 and sqrt(2) scalings make ||Beta|| = ||H||_F, so lstsq's minimum-norm solution is the minimum-Frobenius-norm Hessian (MFN model, Conn–Scheinberg–Vicente) — hence the "fro" in the name. With algorithm_6_4's fully poised 10-point set (n=3) the fit is the unique interpolant, so MFN tie-breaking only matters for smaller/degenerate sets.
- Frame re-centering applied per user: the model fit now happens in the offsets frame. `SmoothFunction.GH` already had `fitfroquad(offsets, f_poised)` (user applied it); added the explanatory comment there, and made the matching change in `NonSmoothFunction.GH`'s gh_type 0 branch (origin-centered `algorithm_6_4` call, `poised = offsets + x` for evaluation/`self.poised`, fit on offsets). `self.poised` stays absolute since solvers compute `poised[best] - x`.
- Why: `fitfroquad` fits q(y) = c + G.y + 0.5 y'Hy in whatever coordinates it is given, so on absolute points the returned G is the slope at the coordinate origin and the model gradient at the iterate is G + Hx — but `bqmin(h, g, -Delta, Delta)` treats g as the slope at x, so the linear term was wrong by Hx (systematic, growing with |x|). Fitting on offsets makes s=0 the iterate: G is the gradient at x and bqmin's box coincides with the trust region. Verified numerically: g_offsets = G_abs + H_abs@x to machine precision (1e-8..1e-13 over 9 configs), and g_offsets tracks a finite-difference gradient of f at x orders of magnitude better than G_abs (e.g. 7.7e-3 vs 5.3e2 at radius 0.01; near Bard poles both degrade, as expected). Methods 0/2 sanity-ran on both classes; method 1 is unaffected (never uses g,h). NOTE: this changes solver behavior — existing 300 logs predate it (left intact; New.txt from tests removed).
